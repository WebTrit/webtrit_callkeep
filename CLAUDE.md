# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

`webtrit_callkeep` is a Flutter VoIP plugin that integrates with the native call management systems (Android Telecom, iOS CallKit). It is structured as a multi-package monorepo.

## Package structure

| Package | Role |
|---|---|
| `webtrit_callkeep/` | Public-facing facade — re-exports platform interface + platform-specific helpers |
| `webtrit_callkeep_platform_interface/` | Abstract contracts (`WebtritCallkeepPlatform`, `CallkeepDelegate`, all models) |
| `webtrit_callkeep_android/` | Android implementation (Kotlin + Pigeon) |
| `webtrit_callkeep_ios/` | iOS implementation (Objective-C + Pigeon) |
| `webtrit_callkeep_linux/`, `_macos/`, `_web/`, `_windows/` | Stub implementations |
| `webtrit_callkeep/example/` | Example app (Flutter + BLoC + freezed) |

## Commands

Run all commands from the relevant package directory.

```bash
# Analyze
flutter analyze

```

### Regenerate Pigeon bindings

After editing `pigeons/callkeep.messages.dart` in either the Android or iOS package:

```bash
flutter pub run pigeon --input pigeons/callkeep.messages.dart
```

This regenerates:
- `lib/src/common/callkeep.pigeon.dart` (Dart side)
- `android/src/main/kotlin/com/webtrit/callkeep/Generated.kt` (Android)
- `ios/Classes/Generated.h` / `ios/Classes/Generated.m` (iOS)

### Example app code generation (freezed)

```bash
# From webtrit_callkeep/example/
dart run build_runner build --delete-conflicting-outputs
```

## Architecture

### Layer model

```
App code
  └── Callkeep (webtrit_callkeep/lib/src/callkeep.dart)   ← main user-facing singleton
        └── WebtritCallkeepPlatform.instance               ← platform interface
              ├── WebtritCallkeepAndroid  (Android)
              └── WebtritCallkeep         (iOS)
```

### Public API surface (webtrit_callkeep)

The main Dart-side entry points are all **singletons**:

| Class | Purpose |
|---|---|
| `Callkeep` | Core call management: setUp/tearDown, report calls, answer/end/hold/mute/DTMF |
| `BackgroundSignalingBootstrapService` | Configure and start/stop the persistent Android background signaling service |
| `BackgroundSignalingService` | Trigger incoming/end calls from the background isolate (Android only) |
| `BackgroundPushNotificationBootstrapService` | Configure push-notification-triggered call handling (Android only) |
| `BackgroundPushNotificationService` | Trigger calls from push notification isolate (Android only) |
| `CallkeepConnections` | Query / clean active call connections |
| `CallkeepPermissions` | Check and request Android-specific permissions |
| `CallkeepLogs` | Set a log delegate to receive native-side logs |
| `CallkeepSound` | Play/stop ringback tone |

### Flutter ↔ Native IPC: Pigeon

All Dart↔Kotlin and Dart↔ObjC communication is code-generated by Pigeon:

- **`@HostApi`** — Dart calls Native (e.g., `PHostApi`, `PHostPermissionsApi`)
- **`@FlutterApi`** — Native calls Dart (e.g., `PDelegateFlutterApi`, `PDelegateBackgroundRegisterFlutterApi`)
- **Converter extensions** — `lib/src/common/converters.dart` in each platform package maps `P*` Pigeon types to `Callkeep*` platform-interface types.

### Android architecture

See `webtrit_callkeep_android/CLAUDE.md` for the full Android-specific architecture (two-process model, service details, IPC bus, boot resilience).

Key points:
- Two OS processes: main app + `:callkeep_core` (houses `PhoneConnectionService`)
- Cross-process IPC via local broadcasts (`ConnectionServicePerformBroadcaster`)
- `ForegroundService` is the main bridge between Telecom events and Flutter (bound to activity)
- Background isolates for signaling (`SignalingIsolateService`) and push notifications (`IncomingCallService`)
- `StorageDelegate` (SharedPreferences) persists config across process restarts

### iOS architecture

- Native side is **Objective-C** (`ios/Classes/*.m`), not Swift
- iOS CallKit requires **UUID**-based call IDs; the Dart layer maps app `callId` strings to UUID v5 deterministically via `_UUIDToCallIdMapping` in `webtrit_callkeep_ios.dart`
- `_CallkeepActionHistory` stores answer/end actions received before the app state initializes, to handle race conditions when notification action buttons are pressed while the app is backgrounded
